## Known Deviations from Spec

These are accepted deviations documented during reviews:

1. **4E schema diverged from plan** — Different column names (`team1_id` not `team_1_id`), `series_format TEXT` instead of `series_config JSONB` (later added via migration 00022). Accepted and patched.
2. **Standings are team-only** — `player_id`/`registration_id` dropped from standings_entries. Individual player standings unsupported.
3. **Rate limiter is in-memory** — Not Redis-backed as planned. Works for single-instance; needs Redis for multi-instance.
4. **External API surface is minimal** — Only `/health` exposed. Tournament/league/match read endpoints not yet built.
5. **API key scoping is flat array** — Plan had hierarchical scope+scope_id. Simpler model shipped.
6. **Pool-play/pool-to-bracket bracket formats** — Not yet implemented in BracketService.

### Cross-cutting contract defects (critical, not in prior remediation scope)

These predate the remediation but block Phase 4+ consumers from reading event / timeline data correctly.

**CR-1 — Event type casing inconsistent across writer, reader, frontend contract.**
- Backend writers emit lowercase snake_case: `"point_team1"`, `"point_team2"`, `"side_out"`, `"remove_point"`, `"confirm_game_over"`, `"confirm_match_over"`, `"timeout"`, `"pause"`, `"resume"`, `"forfeit"`, `"start_game"`, `"undo"`
- Override writer uses mixed case: `"SCORE_OVERRIDE"` (`service/match.go:1718`)
- `enrichedMatchResponse` reader queries `"TIMEOUT_CALLED"` uppercase (`service/match.go:313`) — never matches anything
- Frontend `EventType` union at `types.ts:13-29` uses SCREAMING_SNAKE_CASE (`MATCH_STARTED`, `POINT_SCORED`, `TIMEOUT_CALLED`, etc.)
- Consequence: `EVENT_META` lookup in `EventsTimeline.tsx:254` returns `undefined` for nearly every row → default icon, default grey border, raw `event_type` label. Color-coded borders ship has no visual effect. Timeout counter (C6 partial) never matches. Any frontend logic branching on event type is dead.

**CR-2 — `MatchEventResponse` wire shape mismatch.**
- Frontend `MatchEvent` at `types.ts:94-104` declares `{timestamp, payload, score_snapshot}`
- Backend `MatchEventResponse` at `service/match.go:139-154, 373-400` emits `{created_at, payload, flat team_1_score/team_2_score/set_scores/current_set/current_game/serving_team/server_number}` — **no `timestamp`, no nested `score_snapshot`**
- `EventsTimeline.tsx:287` reads `e.timestamp` → `undefined` → `formatEventTime` breaks
- `EventsTimeline.tsx:316-324` checks `e.score_snapshot` → always `undefined` → expanded view empty

**CR-3 — Timeout event type typo (single line).**
- `CallTimeout` writes `"timeout"` at `service/match.go:1448`
- Enrichment reader queries `"TIMEOUT_CALLED"` at `service/match.go:313`
- Result: `team_X_timeouts_used` always 0 on every response

**CR-4 — Double enrichment on scoring hot path** (code quality reviewer's catch).
- `broadcastMatchUpdate` at `service/match.go:36` calls `enrichedMatchResponse` internally
- Every scoring action (`ScorePoint`, `SideOut`, `RemovePoint`, `ConfirmGameOver`, `ConfirmMatchOver`, `CallTimeout`, `Pause`, `Resume`, `DeclareForfeit`, `Undo`, `OverrideScore`, `StartMatch`, `UpdateStatus`) broadcasts then returns enriched result — same match runs 5+ sub-queries twice = **10+ extra DB round-trips per point scored**, no batching/caching
- At 30 points/game × 3 games × 12 courts ≈ 1,080 mutations per tournament ≈ 10k+ extra queries
